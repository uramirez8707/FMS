19 !> @defgroup diag_manager_mod diag_manager_mod
20 !> @ingroup diag_manager
21 !! @brief diag_manager_mod is a set of simple calls for parallel diagnostics
22 !! on distributed systems. It is geared toward the writing of data in netCDF
23 !! format. See @ref diag_manager for diag table information.
24 !! @author Matt Harrison, Giang Nong, Seth Underwood
25 !!
26 !! <TT>diag_manager_mod</TT> provides a convenient set of interfaces for
27 !! writing data to disk. It is built upon the parallel I/O interface of FMS
28 !! code <TT>/shared/mpp/mpp_io.F90</TT>.
29 !!
30 !! A single group of calls to the <TT>diag_manager_mod</TT> interfaces
31 !! provides data to disk at any number of sampling and/or averaging intervals
32 !! specified at run-time. Run-time specification of diagnostics are input
33 !! through the diagnostics table.
34 !!
35 !! <H4>Usage</H4>
36 !! Use of <TT>diag_manager</TT> includes the following steps:
37 !! <OL>
38 !! <LI> Create diag_table as described in the @ref diag_table_mod
39 !! documentation.</LI>
40 !! <LI> Call @ref diag_manager_init to initialize
41 !! diag_manager_mod.</LI>
42 !! <LI> Call @ref register_diag_field to register the field to be
43 !! output.
44 !! <B>NOTE:</B> ALL fields in diag_table should be registered <I>BEFORE</I>
45 !! the first send_data call</LI>
46 !! <LI> Call @ref send_data to send data to output fields </LI>
47 !! <LI> Call @ref diag_manager_end to exit diag_manager </LI>
48 !! </OL>
49 !!
50 !! <H4>Features</H4>
51 !! Features of <TT>diag_manager_mod</TT>:
52 !! <OL>
53 !! <LI> Ability to output from 0D arrays (scalars) to 3D arrays.</LI>
54 !! <LI> Ability to output time average of fields that have time dependent
55 !! mask.</LI>
56 !! <LI> Give optional warning if <TT>register_diag_field</TT> fails due to
57 !! misspelled module name or field name.</LI>
58 !! <LI> Check if a field is registered twice.</LI>
59 !! <LI> Check for duplicate lines in diag_table. </LI>
60 !! <LI> @ref diag_table_mod can contain fields
61 !! that are NOT written to any files. The file name in diag_table of
62 !! these fields is <TT>null</TT>.</LI>
63 !! <LI> By default, a field is output in its global grid. The user can now
64 !! output a field in a specified region. See
65 !! @ref send_data for more details.</LI>
66 !! <LI> To check if the diag table is set up correctly, user should set
67 !! <TT>debug_diag_manager=.true.</TT> in diag_manager namelist, then
68 !! the the content of diag_table is printed in stdout.</LI>
69 !! <LI> New optional format of file information in @ref diag_table_mod.
70 !! It is possible to have just
71 !! one file name and reuse it many times. A time string will be appended to
72 !! the base file name each time a new file is opened. The time string can be
73 !! any combination from year to second of current model time.
74 !!
75 !! Here is an example file line: <BR />
76 !! <PRE>"file2_yr_dy%1yr%3dy",2,"hours",1,"hours","Time", 10, "days", "1 1 7 0 0 0", 6, "hours"</PRE>
77 !! <BR />
78 !!
79 !! From left to right we have:
80 !! <UL>
81 !! <LI>file name</LI>
82 !! <LI>output frequency</LI>
83 !! <LI>output frequency unit</LI>
84 !! <LI>Format (should always be 1)</LI>
85 !! <LI>time axis unit</LI>
86 !! <LI>time axis name</LI>
87 !! <LI>frequency for creating new file</LI>
88 !! <LI>unit for creating new file</LI>
89 !! <LI>start time of the new file</LI>
90 !! <LI>file duration</LI>
91 !! <LI>file duration unit.</LI>
92 !! </UL>
93 !! The 'file duration', if absent, will be equal to frequency for creating a new file.
94 !!
95 !! Thus, the above means: create a new file every 10 days, each file will last 6 hours from creation time, no files will
96 !! be created before time "1 1 7 0 0 0".
97 !!
98 !! In this example the string
99 !! <TT>10, "days", "1 1 7 0 0 0", 6, "hours"</TT> is optional.
100 !!
101 !! Keywords for the time string suffix is
102 !! <TT>%xyr,%xmo,%xdy,%xhr,%xmi,%xsc</TT> where <TT>x</TT> is a
103 !! mandatory 1 digit number specifying the width of field used in
104 !! writing the string</LI>
105 !! <LI> New time axis for time averaged fields. Users can use a namelist option to handle the time value written
106 !! to time axis for time averaged fields.
107 !!
108 !! If <TT>mix_snapshot_average_fields=.true.</TT> then a time averaged file will have time values corresponding to
109 !! ending time_bound e.g. January monthly average is labeled Feb01. Users can have both snapshot and averaged fields in
110 !! one file.
111 !!
112 !! If <TT>mix_snapshot_average_fields=.false.</TT> The time value written to time axis for time averaged fields is the
113 !! middle on the averaging time. For example, January monthly mean will be written at Jan 16 not Feb 01 as
114 !! before. However, to use this new feature users should <B>separate</B> snapshot fields and time averaged fields in
115 !! <B>different</B> files or a fatal error will occur.
116 !!
117 !! The namelist <B>default</B> value is <TT>mix_snapshot_average_fields=.false.</TT></LI>
118 !! <LI> Time average, Root Mean Square, Max and Min, and diurnal. In addition to time average users can also get then Root Mean Square, Max or Min value
119 !! during the same interval of time as time average. For this purpose, in the diag table users must replace
120 !! <TT>.true.</TT> or <TT>.false.</TT> by <TT>rms</TT>, <TT>max</TT> or <TT>min</TT>. <B><I>Note:</I></B> Currently, max
121 !! and min are not available for regional output.
122 !!
123 !! A diurnal average or the average of an integer power can also be requested using <TT>diurnal##</TT> or <TT>pow##</TT> where
124 !! <TT>##</TT> are the number of diurnal sections or integer power to average.</LI>
125 !! <LI> <TT>standard_name</TT> is added as optional argument in @ref register_diag_field. </LI>
126 !! <LI>When namelist variable <TT>debug_diag_manager = .true.</TT> array
127 !! bounds are checked in @ref send_data.</LI>
128 !! <LI>Coordinate attributes can be written in the output file if the
129 !! argument "<TT>aux</TT>" is given in @ref diag_axis_mod#diag_axis_init . The
130 !! corresponding fields (geolat/geolon) should also be written to the
131 !! same file.</LI>
132 !! </OL>

264  !> @brief Send data over to output fields.
265  !!
266  !> <TT>send_data</TT> is overloaded for fields having zero dimension
267  !! (scalars) to 3 dimension. <TT>diag_field_id</TT> corresponds to the id
268  !! returned from a previous call to <TT>register_diag_field</TT>. The field
269  !! array is restricted to the computational range of the array. Optional
270  !! argument <TT>is_in</TT> can be used to update sub-arrays of the entire
271  !! field. Additionally, an optional logical or real mask can be used to
272  !! apply missing values to the array.
273  !!
274  !! If a field is declared to be <TT>mask_variant</TT> in
275  !! <TT>register_diag_field</TT> logical mask should be mandatory.
276  !!
277  !! For the real mask, the mask is applied if the mask value is less than
278  !! 0.5.
279  !!
280  !! By default, a field will be written out entirely in its global grid.
281  !! Users can also specify regions in which the field will be output. The
282  !! region is specified in diag-table just before the end of output_field
283  !! replacing "none".
284  !!
285  !! For example, by default:
286  !!
287  !! "ocean_mod","Vorticity","vorticity","file1","all",.false.,"none",2
288  !!
289  !! for regional output:
290  !!
291  !! "ocean_mod","Vorticity","vorticity_local","file2","all",.false.,"0.5 53.5 -89.5 -28.5 -1 -1",2
292  !!
293  !! The format of a region is "<TT>xbegin xend ybegin yend zbegin zend</TT>".
294  !! If it is a 2D field use (-1 -1) for (zbegin zend) as in the example above.
295  !! For a 3D field use (-1 -1) for (zbegin zend) when you want to write the
296  !! entire vertical extent, otherwise specify real coordinates. The units
297  !! used for region are the actual units used in grid_spec.nc (for example
298  !! degrees for lat, lon). <B><I>NOTE:</I></B> A FATAL error will occur if
299  !! the region's boundaries are not found in grid_spec.nc.
300  !!
301  !! Regional output on the cubed sphere grid is also supported. To use regional
302  !! output on the cubed sphere grid, first the grid information needs to be sent to
303  !! <TT>diag_manager_mod</TT> using the @ref diag_grid#diag_grid_init subroutine.
304  !!
305  !! @note When using regional output the files containing regional
306  !! outputs should be different from files containing global (default) output.
307  !! It is a FATAL error to have one file containing both regional and global
308  !! results. For maximum flexibility and independence from PE counts one file
309  !! should contain just one region.
310  !!
311  !!
312  !! Time averaging is supported in regional output.
313  !!
314  !! Physical fields (written in "physics windows" of atmospheric code) are
315  !! fully supported for regional outputs.
316  !!
317  !! <B><I>NOTE:</I></B> Most fields are defined in the data domain but use the
318  !! compute domain. In <TT>send_data</TT> the field can be passed in either
319  !! the data domain or in the compute domain. If the data domain is used, the
320  !! start and end indicies of the compute domain (isc, iec, . . .) should be
321  !! passed. If the compute domain is used no indices are needed. The indices
322  !! are for determining halo exclusively. If users want to output the field
323  !! partially they should use regional output as mentioned above.
324  !!
325  !! Weight in Time averaging is now supported, each time level may have a
326  !! different weight. The default of weight is 1.
