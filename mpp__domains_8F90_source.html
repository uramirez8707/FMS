20 !> @defgroup mpp_domains_mod mpp_domains_mod
21 !> @ingroup mpp
22 !> @brief Domain decomposition and domain update for message-passing codes
23 !> @author V. Balaji SGI/GFDL Princeton University
24 !!
25 !> A set of simple calls for domain
26 !! decomposition and domain updates on rectilinear grids. It requires the
27 !! module mpp.F90, upon which it is built.\n
28 !! Scalable implementations of finite-difference codes are generally
29 !! based on decomposing the model domain into subdomains that are
30 !! distributed among processors. These domains will then be obliged to
31 !! exchange data at their boundaries if data dependencies are merely
32 !! nearest-neighbour, or may need to acquire information from the global
33 !! domain if there are extended data dependencies, as in the spectral
34 !! transform. The domain decomposition is a key operation in the
35 !! development of parallel codes.\n
36 !!\n
37 !! mpp_domains_mod provides a domain decomposition and domain
38 !! update API for rectilinear grids, built on top of the mpp_mod API for message passing.
39 !! Features of mpp_domains_mod include:\n
40 !!\n
41 !! Simple, minimal API, with free access to underlying API for more complicated stuff.\n
42 !!\n
43 !! Design toward typical use in climate/weather CFD codes.\n
44 !!
45 !> @par[Domains]
46 !! It is assumed that domain decomposition will mainly be in 2
47 !! horizontal dimensions, which will in general be the two
48 !! fastest-varying indices. There is a separate implementation of 1D
49 !! decomposition on the fastest-varying index, and 1D decomposition on
50 !! the second index, treated as a special case of 2D decomposition, is
51 !! also possible. We define domain as the grid associated with a <I>task</I>.
52 !! We define the compute domain as the set of gridpoints that are
53 !! computed by a task, and the data domain as the set of points
54 !! that are required by the task for the calculation. There can in
55 !! general be more than 1 task per PE, though often
56 !! the number of domains is the same as the processor count. We define
57 !! the global domain as the global computational domain of the
58 !! entire model (i.e, the same as the computational domain if run on a
59 !! single processor). 2D domains are defined using a derived type domain2D,
60 !! constructed as follows (see comments in code for more details).
61 !!
62 !! type, public :: domain_axis_spec
63 !! private
64 !! integer :: begin, end, size, max_size
65 !! logical :: is_global
66 !! end type domain_axis_spec

69 !! private
70 !! type(domain_axis_spec) :: compute, data, global, active
71 !! logical :: mustputb, mustgetb, mustputf, mustgetf, folded
72 !! type(domain1D), pointer, dimension(:) :: list

773  !> @brief Set up a domain decomposition.
774  !!
775  !> There are two forms for the \e mpp_define_domains call. The 2D version is generally
776  !! to be used but is built by repeated calls to the 1D version, also provided.
777  !!
778  !! <br>Example usage:
779  !!
780  !! call mpp_define_domains( global_indices, ndivs, domain, &
781  !! pelist, flags, halo, extent, maskmap )
782  !! call mpp_define_domains( global_indices, layout, domain, pelist, &
783  !! xflags, yflags, xhalo, yhalo, &
784  !! xextent, yextent, maskmap, name )
785  !!
786  !! @param global_indices Defines the global domain.
787  !! @param ndivs The number of domain divisions required.
788  !! @param [inout] domain Holds the resulting domain decomposition.
789  !! @param pelist List of PEs to which the domains are to be assigned.
790  !! @param flags An optional flag to pass additional information
791  !! about the desired domain topology. Useful flags in a 1D decomposition
792  !! include <TT>GLOBAL_DATA_DOMAIN</TT> and
793  !! <TT>CYCLIC_GLOBAL_DOMAIN</TT>. Flags are integers: multiple flags may
794  !! be added together. The flag values are public parameters available by
795  !! use association.
796  !! @param halo Width of the halo.
797  !! @param extent Normally <TT>mpp_define_domains</TT> attempts
798  !! an even division of the global domain across <TT>ndivs</TT>
799  !! domains. The <TT>extent</TT> array can be used by the user to pass a
800  !! custom domain division. The <TT>extent</TT> array has <TT>ndivs</TT>
801  !! elements and holds the compute domain widths, which should add up to
802  !! cover the global domain exactly.
803  !! @param maskmap Some divisions may be masked
804  !! (<TT>maskmap=.FALSE.</TT>) to exclude them from the computation (e.g
805  !! for ocean model domains that are all land). The <TT>maskmap</TT> array
806  !! is dimensioned <TT>ndivs</TT> and contains <TT>.TRUE.</TT> values for
807  !! any domain that must be <I>included</I> in the computation (default
808  !! all). The <TT>pelist</TT> array length should match the number of
809  !! domains included in the computation.
810  !!
811  !! <br>Example usage:
812  !! @code{.F90}
813  !! call mpp_define_domains( (/1,100/), 10, domain, &
814  !! flags=GLOBAL_DATA_DOMAIN+CYCLIC_GLOBAL_DOMAIN, halo=2 )
815  !! @endcode
816  !!
817  !! defines 10 compute domains spanning the range [1,100] of the global
818  !! domain. The compute domains are non-overlapping blocks of 10. All the data
819  !! domains are global, and with a halo of 2 span the range [-1:102]. And
820  !! since the global domain has been declared to be cyclic,
821  !! domain(9)%next => domain(0) and domain(0)%prev =>
822  !! domain(9). A field is allocated on the data domain, and computations proceed on
823  !! the compute domain. A call to mpp_update_domains would fill in the values
824  !! in the halo region:<br>
825  !!<br>
826  !! @code{.F90}
827  !! call mpp_get_data_domain( domain, isd, ied ) !returns -1 and 102
828  !! call mpp_get_compute_domain( domain, is, ie ) !returns (1,10) on PE 0 ...
829  !! allocate( a(isd:ied) )
830  !! do i = is,ie
831  !! a(i) = &lt;perform computations&gt;
832  !! end do
833  !! call mpp_update_domains( a, domain )
834  !! @endcode
835  !!<br>
836  !! The call to mpp_update_domainsfills in the regions outside
837  !! the compute domain. Since the global domain is cyclic, the values at
838  !! \e i=(-1,0) are the same as at \e i=(99,100); and \e i=(101,102)
839  !! are the same as \e i=(1,2).
840  !!
841  !! The 2D version is just an extension of this syntax to two dimensions.
842  !!
843  !! The 2D version of the above should generally be used in
844  !! codes, including 1D-decomposed ones, if there is a possibility of
845  !! future evolution toward 2D decomposition. The arguments are similar to
846  !! the 1D case, except that now we have optional arguments
847  !! flags, halo, extent and maskmap along two axes.
848  !!
849  !! flags can now take an additional possible value to fold one or more edges.
850  !! This is done by using flags \e FOLD_WEST_EDGE, \e FOLD_EAST_EDGE, \e FOLD_SOUTH_EDGE or
851  !! \e FOLD_NORTH_EDGE. When a fold exists (e.g cylindrical domain),
852  !! vector fields reverse sign upon
853  !! crossing the fold. This parity reversal is performed only in the vector version of
854  !! mpp_update_domains. In addition, shift operations may need to be applied to vector fields on
855  !! staggered grids, also described in the vector interface to mpp_update_domains.
856  !!
857  !! <TT>name</TT> is the name associated with the decomposition,
858  !! e.g <TT>'Ocean model'</TT>. If this argument is present,
859  !! <TT>mpp_define_domains</TT> will print the domain decomposition
860  !! generated to <TT>stdlog</TT>.
861  !!
862  !! <br>Examples:
863  !! call mpp_define_domains( (/1,100,1,100/), (/2,2/), domain, xhalo=1 )
864  !! will create the following domain layout:<br>
865  !!<br>
866  !! | domain |domain(1)|domain(2) |domain(3) |domain(4) |
867  !! |--------------|---------|-----------|-----------|-------------|
868  !! |Compute domain|1,50,1,50|51,100,1,50|1,50,51,100|51,100,51,100|
869  !! |Data domain |0,51,1,50|50,101,1,50|0,51,51,100|50,101,51,100|
870  !!
871  !!
872  !! Again, we allocate arrays on the data domain, perform computations
873  !! on the compute domain, and call mpp_update_domains to update the halo region.
874  !!

876  !! we could use:

884  !! |Compute domain|1,100,1,25|1,100,26,50|1,100,51,75|1,100,76,100|
885  !! |Data domain |0,101,1,25|0,101,26,50|0,101,51,75|1,101,76,100|

933 !> Performs halo updates for a given domain.<br>
934 !!
935 !! Used to perform a halo update of a
936 !! domain-decomposed array on each PE. \e MPP_TYPE can be of type
937 !! complex, integer, logical or real of 4-byte or 8-byte kind; of rank up to 5.
938 !! The vector version (with two input data fields) is only present for real types.
939 !! For 2D domain updates, if there are halos present along both
940 !! x and y, we can choose to update one only, by specifying \e flags=XUPDATE or \e flags=YUPDATE.
941 !! In addition, one-sided updates can be performed by setting flags
942 !! to any combination of WUPDATE, EUPDATE, SUPDATE and NUPDATE
943 !! to update the west, east, north and south halos respectively.
944 !! Any combination of halos may be used by adding the requisite flags, e.g:
945 !! \e flags=XUPDATE+SUPDATE or \e flags=EUPDATE+WUPDATE+SUPDATE will update the east,
946 !! west and south halos.<br>
947 !!<br>
948 !! If a call to \e mpp_update_domains involves at least one E-W
949 !! halo and one N-S halo, the corners involved will also be updated, i.e,
950 !! in the example above, the SE and SW corners will be updated.<br>
951 !! If \e flags is not supplied, that is equivalent to \e flags=XUPDATE+YUPDATE.<br>
952 !!<br>
953 !! The vector version is passed the \e x and \e y components of a vector field in tandem,
954 !! and both are updated upon return. They are passed together to treat parity issues on various
955 !! grids. For example, on a cubic sphere projection, the \e x \e y components may be
956 !! interchanged when passing from an equatorial cube face to a polar face.
957 !! For grids with folds, vector components change sign on crossing the fold. Paired scalar
958 !! quantities can also be passed with the vector version if \e flags=SCALAR_PAIR, in which
959 !! case components are appropriately interchanged, but signs are not.<br>
960 !!<br>
961 !! Special treatment at boundaries such as folds is also required for
962 !! staggered grids. The following types of staggered grids are
963 !! recognized:<br>
964 !!<br>
965 !! 1) AGRID: values are at grid centers.<br>
966 !! 2) BGRID_NE: vector fields are at the NE vertex of a grid
